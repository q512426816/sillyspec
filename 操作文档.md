# SillySpec v2.1 操作文档

> **版本**: v2.1.0 | 2026-03-21
> **类型**: 15 个真实的 Claude Code Slash Commands
> **基于**: Superpowers + OpenSpec + GSD 真实源码
> **新增**: 多项目工作区支持（前后端分离、小程序等场景）

---

## 一、这是什么？

SillySpec 是 15 个可以直接在 Claude Code 里用的 slash commands，融合了三个顶级 AI 编码工具的核心能力：

| 来源 | 借鉴能力 | GitHub ⭐ |
|---|---|---|
| **Superpowers** | TDD 纪律、子代理并行开发、精准规划 | 99.6K |
| **OpenSpec** | 变更管理、规范沉淀、自由思考模式 | 32.5K |
| **GSD** | 棕地扫描（7 份文档）、绿地初始化、状态恢复 | 36.4K |

**不是参考文档，是真命令。** 在 Claude Code 里输入 `/sillyspec:` 就能看到完整列表。

## 三、安装

### 方式一：一键远程安装（推荐）

```bash
# 单项目模式
curl -fsSL https://raw.githubusercontent.com/q512426816/sillyspec/main/init.sh | bash

# 多项目工作区模式
curl -fsSL https://raw.githubusercontent.com/q512426816/sillyspec/main/init.sh | bash -s -- --workspace

# 指定目录
curl -fsSL https://raw.githubusercontent.com/q512426816/sillyspec/main/init.sh | bash -s -- --dir ./myapp --workspace
```

### 方式二：本地安装

```bash
cd your-project
bash /path/to/sillyspec/scripts/init.sh
```

安装后重新打开终端，启动 Claude Code：

```bash
claude --dangerously-skip-permissions
```

> 💡 **推荐使用跳过权限模式。** SillySpec 的命令会频繁执行 `git commit`、文件读写、校验脚本等操作，停下来批准会失去意义。这是 GSD 和 OpenSpec 的预期使用方式。

启动后输入 `/sillyspec:` 就能看到 15 个命令。

### 安装参数

| 参数 | 说明 |
|---|---|
| `--workspace` / `-w` | 启用多项目工作区模式 |
| `--dir <path>` / `-d <path>` | 指定安装目录（默认当前目录） |

### 单项目模式 — 安装后目录

```
your-project/
├── .claude/commands/sillyspec/    # 15 个 slash commands
├── .claude/skills/sillyspec/      # 全局 skill（自动加载）
├── .planning/                      # 项目状态和上下文
│   ├── codebase/                  # 代码库扫描文档（棕地）
│   └── phases/                    # 阶段上下文
├── docs/sillyspec/
│   ├── specs/                     # 设计文档
│   └── plans/                     # 实现计划
└── openspec/
    ├── changes/                   # 进行中的变更
    │   └── archive/               # 已归档变更
    └── specs/
```

### 工作区模式 — 安装后目录

```
workspace/
├── .claude/commands/sillyspec/    # 15 个 slash commands
├── .claude/skills/sillyspec/      # 全局 skill
├── .sillyspec/
│   ├── config.yaml                # 🔑 工作区配置（子项目声明）
│   ├── shared/                    # 跨项目共享规范
│   └── workspace/                 # 工作区汇总文档
├── .sillyspec/                     # 项目状态和上下文
├── docs/sillyspec/                # 设计文档和计划
├── openspec/                      # 变更管理
├── frontend/                      # 前端项目（各自独立）
│   └── .sillyspec/codebase/        # 前端扫描文档
├── backend/                       # 后端项目
│   └── .sillyspec/codebase/        # 后端扫描文档
└── miniprogram/                   # 小程序项目
    └── .sillyspec/codebase/        # 小程序扫描文档
```

### 可靠性保障

关键命令内置了双重保障：

- **锚定确认** — propose/plan/execute/verify 执行前强制读取并确认规范文件，防止 AI 跳过上下文
- **Hard Gate 自检** — propose/plan/execute/scan 完成后自动检查输出格式，不通过则修正后再展示
- **校验脚本** — 可用 shell 脚本独立验证 AI 输出：
  ```bash
  bash scripts/validate-all.sh                          # 综合校验
  bash scripts/validate-proposal.sh .sillyspec/changes/xxx # 校验 propose 输出
  bash scripts/validate-scan.sh .planning/codebase       # 校验 7 份扫描文档
  ```

## 四、多项目工作区

### 适用场景

你的项目前后端分离，或者包含多个子项目（前端、后端、小程序、Admin 后台等），各在不同目录甚至不同仓库。

### 配置 config.yaml

```yaml
projects:
  frontend:
    path: ./frontend
    role: 前端 - Vue3 + TypeScript
  backend:
    path: ./backend
    role: 后端 - Node.js + PostgreSQL
  miniprogram:
    path: ./miniprogram
    role: 小程序 - Taro

shared:
  - api-contract.md
  - data-models.md
```

### 工作区命令

```bash
# 添加子项目
/sillyspec:workspace add

# 移除子项目
/sillyspec:workspace remove <name>

# 查看工作区状态
/sillyspec:workspace status

# 查看详细信息
/sillyspec:workspace info
```

### 工作区下的流程

| 命令 | 工作区行为 |
|---|---|
| `/sillyspec:scan` | 遍历所有子项目分别扫描，生成各自的文档 + 工作区汇总 |
| `/sillyspec:scan frontend` | 只扫描指定子项目 |
| `/sillyspec:init` | 提示选择要初始化哪个子项目 |
| `/sillyspec:brainstorm "想法"` | 询问需求属于哪个子项目，加载对应上下文 |
| `/sillyspec:plan` | 任务标注所属项目 `[frontend]`、`[backend]` |
| `/sillyspec:execute` | 按标注 cd 到对应子项目目录执行 |
| `/sillyspec:status` | 显示各子项目状态汇总 |

### 共享规范

跨项目的约定（API 契约、数据模型、接口规范等）放在 `.sillyspec/shared/` 目录下。所有子项目的命令都会自动加载共享规范。

## 五、入口选择

### 绿地项目（空目录）

```bash
/sillyspec:init
```

Claude Code 会深度提问：项目目的、核心功能、技术偏好、约束、不在范围内。然后生成：
- `.sillyspec/PROJECT.md` — 项目概述
- `.sillyspec/REQUIREMENTS.md` — 需求文档（P0/P1/P2 分级）
- `.sillyspec/ROADMAP.md` — 阶段路线图

### 棕地项目（已有代码）

```bash
/sillyspec:scan
```

Claude Code 会扫描代码库，生成 7 份结构化文档：
- `STACK.md` — 技术栈
- `ARCHITECTURE.md` — 架构
- `STRUCTURE.md` — 目录结构
- `CONVENTIONS.md` — 代码约定
- `INTEGRATIONS.md` — 集成
- `TESTING.md` — 测试现状
- `CONCERNS.md` — 技术债务和风险

### 自由思考（随时可用）

```bash
/sillyspec:explore "要不要用 PostgreSQL 还是 SQLite"
```

画 ASCII 图、讨论方案、调查代码库。不写代码，只思考。

## 六、完整工作流教程

### 场景 A：棕地项目添加用户登录

#### Step 1: 扫描（棕地项目）

```bash
/sillyspec:scan
```

AI 扫描代码库，生成 7 份文档。你了解项目全貌。

#### Step 2: 探索需求

```bash
/sillyspec:brainstorm "给网站加用户登录"
```

AI 会：
- 读取 `.sillyspec/codebase/` 了解现有约定
- 逐个提问（一次一个）：登录方式？密码策略？第三方登录？
- 提出 2-3 种方案
- 分段展示设计，你逐段确认
- 生成 `.sillyspec/specs/2026-03-20-user-auth-design.md`
- **不写任何代码**

#### Step 3: 生成规范

```bash
/sillyspec:propose user-auth
```

AI 自动生成：
```
.sillyspec/changes/user-auth/
├── proposal.md          # 动机、范围、成功标准
├── specs/requirements.md  # Given/When/Then 用户场景
├── design.md            # 技术方案、文件清单、API 设计
└── tasks.md             # 任务 checkbox 列表
```

你审阅 proposal 和 design。

#### Step 4: 详细计划

```bash
/sillyspec:plan
```

AI 把 tasks.md 展开为：
- 精确文件路径（`src/auth/login.py`）
- 完整代码示例
- 运行命令和预期输出
- Wave 依赖分组

#### Step 5: TDD 执行

```bash
/sillyspec:execute           # 全部执行
/sillyspec:execute wave-1    # 只执行第一个 Wave
/sillyspec:execute task-3    # 只执行某个任务
```

AI 严格执行 TDD：
```
🔴 写失败测试 → 🟢 写最少代码 → 🔵 重构 → ✅ commit
```

每个任务完成后自动两阶段审查（规范合规 → 代码质量）。

#### Step 6: 验证

```bash
/sillyspec:verify
```

AI 对照规范检查、运行测试套件、搜索技术债务标记，输出验证报告。

#### Step 7: 归档

```bash
/sillyspec:archive user-auth
```

变更移到 `.sillyspec/changes/archive/`，规范沉淀。

### 场景 B：工作区模式（前后端分离项目）

#### Step 1: 初始化工作区

```bash
curl -fsSL https://raw.githubusercontent.com/q512426816/sillyspec/main/init.sh | bash -s -- --workspace
```

#### Step 2: 添加子项目

```bash
/sillyspec:workspace add
# 输入项目名、路径、描述，依次添加前端/后端/小程序
```

#### Step 3: 扫描所有项目

```bash
/sillyspec:scan
# 自动遍历所有子项目，生成各自的文档 + 工作区汇总
```

#### Step 4: 针对子项目开发

```bash
/sillyspec:brainstorm "给用户中心加头像上传"
# AI 询问属于哪个子项目，选择后进入该项目的完整流程

/sillyspec:plan
# 计划中任务标注 [frontend] [backend]，执行时自动切到对应目录
```

## 七、其他命令

### 快速模式

```bash
/sillyspec:quick "修复登录页样式"
```

跳过完整流程，直接 TDD 执行。但仍写测试。

### 中断恢复

```bash
/sillyspec:handoff    # 保存状态，下班走人
/sillyspec:resume     # 第二天恢复
/sillyspec:continue   # 自动判断下一步
/sillyspec:status     # 查看进度
```

### 场景速查

| 场景 | 命令 |
|---|---|
| 新项目（空目录） | `/sillyspec:init` |
| 已有项目第一次用 | `/sillyspec:scan` |
| 前后端分离 / 多项目 | `init.sh --workspace` → `/sillyspec:workspace add` |
| 不确定要做什么 | `/sillyspec:explore "想法"` |
| 新功能 | brainstorm → propose → plan → execute → verify → archive |
| bug 修复 | `/sillyspec:quick "修复 xxx"` |
| 改个颜色/文案 | `/sillyspec:quick "改 xxx"` |
| 修 typo | 直接改 |
| 中断恢复 | `/sillyspec:resume` |
| 不知道下一步 | `/sillyspec:continue` |

## 十一、常见问题

### Q: 安装后看不到 `/sillyspec:*` 命令？
重新打开终端。Claude Code 只在启动时加载 `.claude/commands/`。

### Q: scan 扫描大项目很慢？
可以用 `/sillyspec:scan api` 只扫描特定区域。

### Q: execute 的子代理并行能用吗？
取决于你的 Claude Code 是否支持 subagent。不支持时会自动降级为串行执行。

### Q: 工作区模式下，子项目需要是 git 仓库吗？
不需要。`config.yaml` 里的 `path` 可以是任何目录。但建议每个子项目独立 git 管理。

### Q: 工作区和单项目能互相切换吗？
可以。删除 `.sillyspec/config.yaml` 就回到单项目模式。重新创建 config.yaml 则进入工作区模式。

### Q: 和分别装三个工具有什么区别？
| 维度 | 分别装 | SillySpec |
|---|---|---|
| 安装 | 3 次 | 1 个脚本 |
| 命令 | 3 套 | 1 套 `/sillyspec:*` |
| 流程串联 | 手动 | 每个 command 结尾提示下一步 |
| 绿地/棕地 | 各自处理 | `init` / `scan` 统一入口 |
| 多项目 | 不支持 | `workspace` 工作区模式 |
| Token 消耗 | 三套叠加 | 精简 1 套 |

### Q: 可以只用一部分命令吗？
可以。不需要 explore？删掉 `commands/sillyspec/explore.md`。按需裁剪。

### Q: 原版更新了怎么办？
目前手动同步。后续考虑 git submodule 或发布到 skills.sh。
